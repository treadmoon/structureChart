# StructureChart UE 改造落地方案

## 1. 目标与范围

本轮改造目标：将当前 Demo 型交互升级为可上线产品体验，优先解决“可用、可控、可扩展”。

范围：
- 示例页交互框架（控制栏、搜索、菜单、状态反馈）
- 图谱编辑体验（误操作防护、撤销重做、保存）
- 视觉与可访问性基础能力（token、焦点、键盘）

不在本轮范围：
- 后端接口设计
- 权限体系
- 多人协作编辑

## 2. 信息架构（IA）

页面拆分为 4 个区域：

1. 顶部状态栏
- 内容：当前模式（View/Edit）、数据版本状态（未保存/已保存）、最后保存时间
- 目的：让用户随时知道“当前是否在改结构”

2. 左侧工具面板（可折叠）
- 视图工具：Fit、Zoom In、Zoom Out、重置
- 搜索工具：关键词输入、结果列表、结果计数
- 筛选工具：按中心/部门类型过滤

3. 中央图谱画布
- 组织树主视图
- 拖拽重排/改父节点
- 节点选中高亮

4. 右侧详情面板
- 选中节点的基本信息（姓名、职级、部门、ID）
- 快捷操作（添加子节点、编辑、删除）
- 关系预览（上级/直属下级数量）

移动端策略：
- 左右面板折叠为底部抽屉
- 画布优先占满屏

## 3. 关键交互流程

### 3.1 搜索与定位
1. 用户输入关键词（姓名/职位/ID）
2. 下拉出现匹配结果（最多 10 条）
3. 点击结果：自动展开路径 -> 居中 -> 高亮 -> 右侧面板展示详情
4. 无匹配时显示空状态（非 console）

### 3.2 编辑与提交
1. 点击“进入编辑模式”
2. 拖拽节点后进入“草稿变更态”
3. 顶部状态显示“未保存”
4. 用户可执行：撤销 / 重做 / 保存 / 放弃
5. 只有保存后才触发最终 `onDataChange` 提交

### 3.3 节点操作菜单
1. 点击节点菜单按钮
2. 打开结构化菜单（编辑、添加子节点、删除）
3. 删除需二次确认（展示影响提示）
4. 操作结果通过 Toast 反馈

## 4. 视觉与体验规范

### 4.1 设计 Token（先统一，再多主题）
- 字体：统一一套中英文栈
- 间距：4/8/12/16/24 体系
- 圆角：4/8
- 阴影：层级 1/2
- 色彩：主色、成功、警告、危险、文本主次、边框、背景

### 4.2 卡片层级统一
每张卡片固定层级：
1. 顶部：类型与操作
2. 中部：姓名（主信息）
3. 底部：职位/辅助信息（子信息）

### 4.3 交互反馈
- hover：轻微抬升
- selected：描边 + 阴影（不只靠颜色）
- drag target：高对比占位指示
- destructive：红色 + 二次确认

## 5. 可访问性（A11y）基线

最低标准（首轮必须达成）：
- 可键盘访问：Tab 焦点可达主要控件与节点
- 明确焦点态：focus-visible 轮廓
- 菜单与按钮具备 ARIA label
- 颜色对比达到 WCAG AA

## 6. 代码改造清单（按文件映射）

### 6.1 示例应用层

1. `src/example/main.ts`
- 重构控制区为语义化布局（toolbar + panel）
- 新增搜索建议与结果面板
- 新增编辑会话状态（draft/dirty/saved）
- 将菜单行为从 `onclick` 字符串迁移为事件绑定
- 将 `alert` 改为统一反馈组件（toast/confirm）

2. `src/example/style.css`
- 增加布局网格（桌面三栏 / 移动抽屉）
- 增加 token 变量
- 增加 focus-visible 样式与移动端断点

### 6.2 核心图谱层

3. `src/plugin/OrgChart.ts`
- 增加撤销/重做栈（基于 rawData 快照）
- 拖拽变更先写入 draft，不直接提交最终变更
- 增加更丰富事件回调：`onSelectionChange`、`onEditStateChange`
- 增强 `centerNode` 支持搜索结果对象（不仅 ID）

4. `src/plugin/templates.ts`
- 移除内联事件字符串，改 data-attribute + 外部事件代理
- 统一卡片信息层级与可访问性属性

5. `src/plugin/styles.css`
- 提取颜色/间距 token
- 强化选中态与拖拽态可见性
- 增加 focus-visible、reduced-motion 兼容

## 7. 实施节奏（建议）

### Sprint 1（3-5 天）
- 控制区重构
- 搜索升级（姓名/职位/ID）
- 菜单机制重构（去 inline + 去 alert）

### Sprint 2（4-6 天）
- 编辑草稿态
- 撤销/重做
- 保存/放弃流程

### Sprint 3（3-4 天）
- 视觉 token 统一
- A11y 基线
- 移动端适配

## 8. 验收标准（量化）

- 任务成功率：首次定位目标节点成功率 > 95%
- 编辑可靠性：误操作后可恢复率 100%（撤销有效）
- 易用性：新用户在 2 分钟内完成“搜索 + 编辑 + 保存”
- 质量门槛：关键交互有基础用例（搜索、拖拽、保存、撤销）

## 9. 下一步执行建议

建议按以下顺序直接开工：
1. 先做搜索与控制区重构（最直接改善体验）
2. 再做编辑草稿态与撤销重做（降低风险）
3. 最后统一视觉和可访问性（收口）
